### 🛡️ 產線防禦與基礎設施鐵律 (The Ironclad Rules)
* **雙棲實體路徑隔離**：透過 `.env` 檔案中的 `WORKSPACE_ROOT` 變數與 `env_manager.py` 進行跨平台路徑解析。所有大容量 I/O 必須寫入此隔離區。Windows 開發期設定為本地目錄（如 `F:\AI_Workspace\AI_DRAMA_FACTORY`），Mac 部署後強制綁定 `/Volumes/AI_Workspace/` 保護 256GB SSD。
* **標準化退出碼與日誌瘦身 (Log Sanitization)**：所有 Python 腳本在捕捉到致命錯誤時，必須強制呼叫 `sys.exit(1)`。寫入 `project_learning.md` 的錯誤日誌必須經過正則清洗，嚴禁寫入動輒數十行的 Stack Trace，僅保留「Exception Name」與「最後一行關鍵錯誤訊息」，防止檔案無限膨脹導致復盤 Token 暴增。
* **指令注入防禦 (Command Injection Prevention)**：在 `auto_editor.py` 或任何呼叫 FFmpeg 等底層命令的腳本中，嚴禁使用 `shell=True`。所有 Subprocess 必須以陣列形式傳遞參數 `["ffmpeg", "-i", ...]`，並對 LLM 生成的檔名/路徑強制執行安全字元過濾，防止惡意 Prompt 注入導致 `rm -rf /` 災難。
* **原生外部機密隔離 (External Secrets)**：徹底淘汰實體 `.env` 檔案。全面導入 OpenClaw 的「外部機密管理」，透過終端機執行 `openclaw secrets configure` 寫入 API 金鑰。腳本運行時透過 `target-path` 驗證動態調用，實現「零明文落地」的軍規級防護。
* **API 級別結構化輸出 (Structured Outputs)**：捨棄易脆斷的正則表達式清洗。所有呼叫大模型的腳本，必須在 API 請求層面強制啟用 `response_format={ "type": "json_object" }` 或給定嚴格 JSON Schema。從底層強制約束 100% 合法 JSON，將防呆成本降至 $0。
* **原子寫入防護 (Atomic Writes)**：針對憑證檔 (`state.json`) 與狀態池，嚴禁直接 `open(file, 'w')`。必須強制寫入 `.tmp` 暫存檔，寫入成功後再呼叫作業系統層級的 `os.replace()` 覆蓋原檔。確保在斷電或磁碟瞬斷時，JSON 檔案 100% 不會發生 0KB 靜默損毀。
* **分卷與首尾影格繼承 (Chunking & Frame Inheritance)**：將 5~10 分鐘的劇本切割為多個 60 秒的卷宗 (Volumes)。在呼叫 Veo API 時，下一卷的第一個分鏡，必須強制讀取上一卷最後一個分鏡的「最後一幀 (Last Frame)」作為 Image Prompt 參考起點，達成 100% 無縫連貫。
* **API 限流與退避防護 (Rate Limit & Exponential Backoff)**：嚴禁在呼叫 Google Cloud API 遇到 `429 Too Many Requests` 時直接崩潰。所有 API 請求必須包裝在指數退避 (Exponential Backoff) 迴圈中，確保產線能在配額受限時優雅休眠與重試。
* **內容合規護欄 (Safety Guardrails)**：Google Veo 具備嚴格的安全審查。LLM 在生成分鏡 Prompt 時，必須注入系統級過濾指令，嚴禁出現暴力、自殘或血腥字眼，一律以「荒謬、誇張的定格動畫肢體」來隱喻職場壓抑，確保 API 通過率 100%。
* **媒體生命週期管理 (Media Lifecycle Management)**：嚴格執行「大檔案視為消耗品，小檔案視為永久資產」。1小時的 MP4 影片在上傳/推流後 48 小時內，必須強制從本機 SSD 刪除 (TTL=48h)。產線中繼暫存檔 (`temp_*.wav`) 必須在腳本結束時立即抹除 (Garbage Collection)。高商業價值的 WAV 母帶保留 7 天後，必須自動打包轉移至 S3/B2 冷儲存，釋放 Mac mini 空間。
* **Telegram-First 行動控制協議 (Mobile-Centric Control)**：產線所有關鍵決策點 (Checkpoint) 必須透過 Telegram Inline Keyboard (內嵌按鈕) 實現「一鍵化」操作。系統嚴禁要求 CEO 回到電腦端輸入指令。所有的「採用/放棄」或「確認發行」回饋，必須在手機端 1 秒內完成。
* **試聽與配額機制 (Audition & Quota)**：Phase 1 導入「Demo 審核機制」。系統自動截取原始音軌前 60 秒作為試聽檔。每日產線必須達成「10 首 CEO 採用」之目標。若未滿 10 首，`pipeline_runner.py` 需持續運作直至配額達標，確保每日庫存水位穩定。
* **防流程偏移與 API 隔離鐵律 (Pipeline Drift Prevention & API Discipline)：確立 scripts/gear1_prod/pipeline_runner.py 為產線的「唯一最高指揮官與單一入口 (Single Entry Point)」。所有 Suno 音樂生成必須透過雲端 TTAPI 執行，嚴禁在本機維護 Suno 登入態或 Docker 容器。絕對禁止開發者或 AI 代理單獨呼叫底層腳本（如 suno_api_engine.py）。當 TTAPI 回傳 500 錯誤或任務失敗時，必須遵循「ZERO SILENT FAILURES」，寧可停線報警，嚴禁啟動旁路腳本抓取歷史庫存舊檔充數。

## ▌C. v8.6 R&S Echoes 產線工作流 (Lofi Assembly & Mastering Strategy)

### Phase 1 — Suno 節拍生成與 CEO 空投 (The Beat Genesis)
**戰術**：
- **自動模式：** suno_lofi_generator.py 透過 TTAPI 雲端代理，使用 Lofi 專屬 Prompt 批次生成純音樂節拍。下列是自動模式流程:
	- **Demo 審核循環 (Audition Loop)**：AD 腳本產出原始 MP3 後，自動呼叫 FFmpeg 截取 1 分鐘 Demo 並傳送至 Telegram。
	- **一鍵決策**：Telegram 訊息附帶 [✅ 採用] 與 [❌ 放棄] 按鈕。
		- 點擊 [✅ 採用]：檔案移入 `ceo_approved_beats/`，每日計數器 +1。
		- 點擊 [❌ 放棄]：檔案移入垃圾區，系統自動生成/抓取下一首。
	- **目標達成**：每日必須達成 10 首「採用」狀態，產線方可推進至 Phase 2。
- **手動模式 (CEO Dropzone)**：產線支援「人工品管介入」。CEO 於 Suno 網頁版挑選滿意的 MP3，直接存入 `assets/audio/ceo_approved_beats/`，觸發後續產線。

### Phase 2 — 純淨母帶處理 (Naked Auto-Mastering)
**腳本**：`scripts/gear1_prod/audio_mastering_engine.py`
**戰術**：
- **廢除混音疊加 (Zero Foley/SFX)**：為確保 Lofi 音樂最純粹的放鬆聽感，全面停用大自然白噪音與空間混響 (Reverb) 的疊加邏輯。
- **工業級發行規格 (Loudness Normalization)**：僅保留 FFmpeg `-af loudnorm` 濾鏡進行精準響度標準化，強制轉換為 44.1kHz / 24-bit 無損 WAV 格式。
- **雙軌道輸出**：自動輸出 YouTube 專用版 (-14 LUFS) 與 Spotify 商業發行專用版 (-16 LUFS, 真實峰值 -2.0 dBTP)。

### Phase 3 — 視覺錨點與微動態 I2V (The Cinemagraph)
**戰術**：
- 利用 Kling 或 智譜 CogVideoX 生成 5 秒的極靜謐微動態影片 (Cinemagraph)。
- **核心鐵律**：絕對鎖定鏡頭 (Static Camera)。只允許水波流動、光影閃爍、人物呼吸。此影片作為產線的「視覺母帶」，無需頻繁重算，可無限次重複使用以攤平 API 成本。

### Phase 4 — 無縫縫合機 (1-Hour Loop Assembler)
**腳本**：`scripts/gear1_prod/lofi_assembler.py`
**戰術**：
- **畫面無限延伸**：讀取 5 秒的微動態母帶，使用 FFmpeg 濾鏡 (`reverse` 與 `concat`) 執行 Ping-Pong 倒轉法，或 Crossfade 交叉淡化法，將畫面無縫延伸至 1 小時。
- **音軌淡入淡出**：將 Phase 2 處理好的多首 Lofi 節拍，加上 5-10 秒的交叉淡入淡出，與 1 小時的畫面合併，封裝輸出 `R_S_Echoes_VolX_1Hour.mp4`。

### Phase 5 — 數位唱片企劃引擎與授權閘道 (Metadata & Authorization)
**腳本**：`scripts/gear1_prod/music_metadata_engine.py`
**戰術**：
- 呼叫智譜 GLM-4，依據本期音樂的氛圍（如：Midnight Rain, Coffeehouse），生成符合全球音樂發行商標準的 JSON 企劃書。
- 必須包含：高質感英文曲名 (Track Title)、精準曲風分類 (Sub-genre)、以及 YouTube 專用的 SEO 描述。
