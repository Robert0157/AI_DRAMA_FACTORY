### 🛡️ 產線防禦與基礎設施鐵律 (The Ironclad Rules)
* **雙棲實體路徑隔離**：透過 `.env` 檔案中的 `WORKSPACE_ROOT` 變數與 `env_manager.py` 進行跨平台路徑解析。所有大容量 I/O 必須寫入此隔離區。Windows 開發期設定為本地目錄（如 `F:\AI_Workspace\AI_DRAMA_FACTORY`），Mac 部署後強制綁定 `/Volumes/AI_Workspace/` 保護 256GB SSD。
* **標準化退出碼與日誌瘦身 (Log Sanitization)**：所有 Python 腳本在捕捉到致命錯誤時，必須強制呼叫 `sys.exit(1)`。寫入 `project_learning.md` 的錯誤日誌必須經過正則清洗，嚴禁寫入動輒數十行的 Stack Trace，僅保留「Exception Name」與「最後一行關鍵錯誤訊息」，防止檔案無限膨脹導致復盤 Token 暴增。
* **指令注入防禦 (Command Injection Prevention)**：在 `auto_editor.py` 或任何呼叫 FFmpeg 等底層命令的腳本中，嚴禁使用 `shell=True`。所有 Subprocess 必須以陣列形式傳遞參數 `["ffmpeg", "-i", ...]`，並對 LLM 生成的檔名/路徑強制執行安全字元過濾，防止惡意 Prompt 注入導致 `rm -rf /` 災難。
* **原生外部機密隔離 (External Secrets)**：徹底淘汰實體 `.env` 檔案。全面導入 OpenClaw 的「外部機密管理」，透過終端機執行 `openclaw secrets configure` 寫入 API 金鑰。腳本運行時透過 `target-path` 驗證動態調用，實現「零明文落地」的軍規級防護。
* **API 級別結構化輸出 (Structured Outputs)**：捨棄易脆斷的正則表達式清洗。所有呼叫大模型的腳本，必須在 API 請求層面強制啟用 `response_format={ "type": "json_object" }` 或給定嚴格 JSON Schema。從底層強制約束 100% 合法 JSON，將防呆成本降至 $0。
* **原子寫入防護 (Atomic Writes)**：針對憑證檔 (`state.json`) 與狀態池，嚴禁直接 `open(file, 'w')`。必須強制寫入 `.tmp` 暫存檔，寫入成功後再呼叫作業系統層級的 `os.replace()` 覆蓋原檔。確保在斷電或磁碟瞬斷時，JSON 檔案 100% 不會發生 0KB 靜默損毀。
* **分卷與首尾影格繼承 (Chunking & Frame Inheritance)**：將 5~10 分鐘的劇本切割為多個 60 秒的卷宗 (Volumes)。在呼叫 Veo API 時，下一卷的第一個分鏡，必須強制讀取上一卷最後一個分鏡的「最後一幀 (Last Frame)」作為 Image Prompt 參考起點，達成 100% 無縫連貫。
* **API 限流與退避防護 (Rate Limit & Exponential Backoff)**：嚴禁在呼叫 Google Cloud API 遇到 `429 Too Many Requests` 時直接崩潰。所有 API 請求必須包裝在指數退避 (Exponential Backoff) 迴圈中，確保產線能在配額受限時優雅休眠與重試。
* **內容合規護欄 (Safety Guardrails)**：Google Veo 具備嚴格的安全審查。LLM 在生成分鏡 Prompt 時，必須注入系統級過濾指令，嚴禁出現暴力、自殘或血腥字眼，一律以「荒謬、誇張的定格動畫肢體」來隱喻職場壓抑，確保 API 通過率 100%。

## ▌C. v8.2 黃金三角產線工作流 (Chunking & API Strategy)

### Phase 1 — Suno API 總譜生成 (The Epic Master Track)
**腳本**：`scripts/gear1_prod/suno_api_engine.py`
**戰術**：
- **中繼站串接**：腳本不再直連外部 SaaS，而是向本地運行中的 `http://localhost:3000/api/custom_generate` 發送請求。
- **純器樂強制**：參數帶入 `"make_instrumental": true`。
- **輪詢機制**：透過 `/api/get` 輪詢狀態直到 `status == 'streaming'`，穩定下載 5~10 分鐘的純器樂配樂 (`music_epic.mp3`)。

### Phase 2 — 大腦分卷切割 (Chunking Orchestrator)
**腳本**：`scripts/gear1_prod/music_first_orchestrator.py`  
**戰術**：
- 讀取 10 分鐘的 `music_epic.mp3`。
- 使用 `librosa` 抓取節拍，強制每 60 秒截斷一次。
- 呼叫智譜 GLM-4-Flash，輸出 `vol_1.json` 到 `vol_10.json`，每份 JSON 嚴格控制在 12~15 鏡。

### Phase 2.5 — 視覺基準圖生成 (Image Anchor Generation)
**腳本**：`scripts/gear1_prod/midjourney_api_engine.py`  
**戰術**：
- 產線抽出劇本中的核心主角與場景描述，送交 Midjourney API。
- 產出極致電影感、高材質細節的靜態圖 (e.g., `anchor_01.jpg`)，做為後續影片生成的「首幀基準」。

### Phase 3 — Kling API 算圖引擎 (Image-to-Video Engine)
**腳本**：`scripts/gear1_prod/kling_api_engine.py` (取代 Veo)  
**戰術**：
- 讀取 `vol_n.json` 中的各個分鏡。
- **核心鐵律**：將 Phase 2.5 產出的 `anchor_01.jpg` (或上一鏡的最後一幀) 作為 Image Prompt 餵給 Kling API 進行圖生片。借助可靈強大的「肢體一致性」，確保木偶長相不變形。
- 內建 Exponential Backoff 抵抗網路不穩與 API 限流錯誤。

### Phase 4 — 卷宗獨立剪輯與終極縫合 (M4 Concat Assembly)
