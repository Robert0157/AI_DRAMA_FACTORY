**核心戰略升級**：
全面邁向「無頭伺服器 (Headless Server)」運營。以 `rs_manager.py` 與 `telegram_manager_bot.py` 雙軌中控台實現 100% 遠端指揮。導入 Suno「時間膨脹」Prompt 駭客技巧實現極致降本；實裝「50/25/25 智能選曲」、「雙軌 SFX 遮罩」、「自動商業化打包」與「100 首定額冷金庫 FIFO 代謝」，打造具備自我淨化與動態適應能力的工業級產線。

> **The Cost-Optimized Weekly Genesis (成本極致最佳化戰略)**：  
> 產線由「每日高頻打靶」轉向「每週精準供彈」。透過 Suno 提示詞的「雙重刪節號 (...)」時間膨脹技巧，單曲長度突破 7 分鐘，使 1 小時影片所需曲目大幅降低。系統每週僅需生成極少數的「子彈」供 CEO 於網頁端手動打靶。滿意的成品空投回產線後，全面啟動 **Protocol L (音樂資產保鮮庫)**。
> 透過 FFmpeg 的零成本變速/變調引擎，母帶將被進行嚴格的「三振防線」重製 (derivation_count < 3)，配合 50/25/25 排播比例，裂變出足以支撐多頻道的發行量，將 API 成本與手動作業時間壓縮至最低極限。

### 🛡️ 產線防禦與基礎設施鐵律 (The Ironclad Rules)
* **最高修憲權 (CEO Override)**：所有系統參數、音樂風格、排播比例、上架決策，僅 CEO 有權定奪。系統必須無條件執行 CEO 的指令 (`rs_manager.py` 與 Telegram 雙軌授權)。
* **雙棲實體路徑隔離**：透過 `.env` 檔案中的 `WORKSPACE_ROOT` 變數與 `env_manager.py` 進行跨平台路徑解析。所有大容量 I/O 必須寫入此隔離區。Windows 開發期設定為本地目錄，Mac 部署後強制綁定外部硬碟或指定路徑，保護 256GB SSD。
* **標準化退出碼與日誌瘦身 (Log Sanitization)**：所有 Python 腳本在捕捉到致命錯誤時，必須強制呼叫 `sys.exit(1)`。錯誤日誌必須經過清洗，嚴禁寫入動輒數十行的 Stack Trace，僅保留「Exception Name」與「關鍵錯誤訊息」，防止檔案無限膨脹。
* **指令注入防禦 (Command Injection Prevention)**：在任何呼叫 FFmpeg 或 ffprobe 等底層命令的腳本中，嚴禁使用 `shell=True`。所有 Subprocess 必須以陣列形式傳遞參數 `["ffmpeg", "-i", ...]`，防止惡意注入導致系統災難。
* **API 級別結構化輸出 (Structured Outputs)**：捨棄易脆斷的正則表達式清洗。所有呼叫大模型 (如 GLM-4) 的腳本，必須在 API 請求層面強制啟用 `response_format={ "type": "json_object" }` 或給定嚴格 JSON Schema，確保 100% 合法 JSON 輸出。
* **原子寫入防護 (Atomic Writes)**：針對憑證檔、JSON 中繼資料 (`metadata_distrokid.json`) 與資料庫狀態，嚴禁直接 `open(file, 'w')`。必須強制寫入 `.tmp` 暫存檔，寫入成功後再呼叫 `os.replace()` 覆蓋原檔，確保斷電或意外中斷時檔案 100% 不會發生 0KB 靜默損毀。
* **API 限流與退避防護 (Rate Limit & Exponential Backoff)**：呼叫外部 API 遇到限制或超時時，嚴禁直接崩潰。所有 API 請求必須包裝在指數退避迴圈中，確保產線能優雅休眠與重試。
* **媒體生命週期管理 (Media Lifecycle Management)**：嚴格執行「大檔案視為消耗品，小檔案視為永久資產」。1 小時的 MP4 影片與中繼暫存檔 (`temp_*.wav`) 必須在產出發布後盡速清除 (Garbage Collection)。高商業價值的 WAV 母帶依 Protocol L 規則進行代謝或封存。
* **最高修憲權限鎖定 (Constitution Immutability)**：嚴禁工程團隊、AI 輔助工具擅自修改產線憲法！任何關於 `.clinerules`, `copilot-instructions.md`, `project.yml` 或 `架構說明書` 的核心文件更新，**僅限 CEO 或 CTO 手動執行**。
* **防流程偏移與單一入口鐵律 (Pipeline Drift Discipline)**：確立 `rs_manager.py` (本地) 與 `telegram_manager_bot.py` (遠端) 為產線唯二合法入口。嚴禁繞過中控台執行零散腳本。
* **資產永續與三振再製防線 (The Rule of 3 Derivations)**：為避免聽覺疲勞與演算法降權，同一首入庫母帶的「二次利用 (變速、變調等)」，生命週期內絕對禁止超過 3 次。資料庫必須嚴格追蹤 `derivation_count`，達標後依規定進行封存或物理刪除。
* **靶場清空鐵律 (The Clean Range Policy)**：
   - 每次產線執行後，或是 CEO 完成聽審後，必須強制清空 `assets/audio/ceo_approved_beats/` 靶場。
   - 絕不允許同一發子彈（音樂檔案）被重複掃描或重複母帶化。

## ▌C. R&S Echoes 產線工作流 (Lofi Assembly & Mastering Strategy v9.5)

### Phase 1 — 靈感生成與手動空投 (Time-Dilated Genesis & Ingestion)
**戰術**：
- **第一步：自動供彈與時間膨脹 (Prompt Arsenal & Time Dilation)**：產線啟動 `generate_ceo_prompts.py`。呼叫 GLM-4 引擎生成高品質氛圍提示詞，並在結構末端**強制注入「雙重刪節號 `...` `...`」時間膨脹代碼**。這能欺騙 Suno 模型，強制將單曲長度拉長至 7 分鐘以上。
- **第二步：每週精準空投 (Weekly Manual QA)**：捨棄每日高頻打靶。CEO 每週於 Suno 網頁端利用免費額度生成，挑選完美達標（長度 > 7 分鐘、無雜音）的成品，手動下載至 `assets/audio/ceo_approved_beats/` 空投區。
- **第三步：總指揮官盤點 (Pipeline Initiation)**：`pipeline_runner.py` (或透過 Telegram `/build`) 啟動時，優先盤點 `ceo_approved_beats/` 的新子彈準備進入母帶化。
- **第四步：強制清空靶場 (Clean Range Policy)**：當新素材順利進入母帶處理後，系統強制呼叫 `os.remove()` 清空 `ceo_approved_beats/` 目錄，確保靶場歸零，杜絕舊檔重複處理。

### Phase 2 — 嚴格母帶處理 (Strict Auto-Mastering)
**腳本**：`scripts/gear1_prod/audio_mastering_engine.py`
**戰術**：
- **廢除舊版閘門與提前混音**：全面停用舊版 Telegram 審批攔截閘門，並將環境音 (SFX) 疊加作業延後至 Phase 4 處理。此階段專注於「純淨母帶 (Naked Mastering)」的生成。
- **工業級發行規格 (Loudness Normalization)**：僅保留 FFmpeg `-af loudnorm` 濾鏡進行精準響度標準化。統一強制套用 Spotify 商業發行標準 (-16 LUFS, 真實峰值 -2.0 dBTP) 與 44.1kHz/24-bit 無損 WAV 格式。
- **自動入庫 (Vault Deposit)**：母帶處理完畢後，直接移入 `vault_ready_for_mix/` 金庫，成為可供 50/25/25 演算法抽取的正式資產。

### Phase 3 — 發行企劃中繼引擎 (Metadata & SEO Engine)
**腳本**：`scripts/gear1_prod/music_metadata_engine.py`
**戰術**：
- **結構化企劃生成 (Structured Outputs)**：呼叫智譜 GLM-4 Flash 引擎，根據 Lofi 品牌調性自動生成專輯標題 (如 Rhythmic Echoes Vol.X)、曲風標籤與 YouTube SEO 描述。
- **廢除草稿鐵律 (No Drafts Allowed)**：全面刪除舊版的 `YouTube_CheatSheet_草稿` 生成邏輯。本階段**僅允許**輸出純數據的 `metadata_distrokid.json` 作為單一真相來源 (Single Source of Truth)，為後續 Phase 5 的「一站式文案合體」做好準備。

### Phase 4 — 智能選曲與無縫縫合機 (Smart Vault Selection & Audio Assembly)
**腳本**：`scripts/gear1_prod/lofi_assembler.py`
**戰術**：
- **動態時長驅動 (Duration-Driven)**：徹底廢除固定首數（如固定抓 25 首）。系統會先探測金庫庫存的平均時長，動態計算所需數量，精準抓取足夠的歌曲以拼滿 3600 秒 (1 小時)。
- **50/25/25 智能引擎與回寫機制**：嚴格依循「0次新歌 (50%)、1次衍生 (25%)、2次衍生 (25%)」比例抽取，並強制執行「同源排斥」確保 1 小時內不會出現相似基因。被選中的曲目會在組合前立即回寫資料庫，將 `derivation_count` 精準 +1。
- **雙軌 SFX 環境音引擎**：支援大自然環境音（如雨聲、流水）疊加。可選「Transition 模式 (換歌交界 12 秒以 2% 音量浮現，依賴 FFmpeg 單引號精準遮罩)」或「Global 模式 (全域 2% 鋪底)」。最終輸出 1 小時純淨無縫 `.wav` 長曲。

### Phase 5 — 萬能影片循環與一站式打包 (Dynamic Video Loop & Inline Packaging)
**腳本**：`scripts/gear1_prod/pipeline_runner.py` / `video_processor.py`
**戰術**：
- **動態探測循環 (Dynamic Ping-Pong)**：廢除硬編碼影片長度（不再鎖死 5 秒）。呼叫 `ffprobe` 動態讀取 AI 微動態母帶的精準時長（如 4.8 秒或 8.1 秒），執行正播+倒播 (Ping-Pong Seamless) 無縫拼接至 1 小時，並強制拋棄視頻原生音軌，確保 100% 音訊純淨度。
- **一站式商業企劃合體 (In-line CheatSheet)**：廢除繁瑣的「草稿」機制。系統讀取 Phase 3 的 JSON 與 Phase 4 的時間軸，直接於記憶體中進行「商業化修飾」——過濾時間戳記，自動將 `tempo_up` 轉換為 `[Remix 1]`，並為重複曲名動態加上 `Vol. N` 標籤。一氣呵成輸出可直接對外發布的正式版 `YouTube_CheatSheet_[時間戳].txt`。

### Phase 6 — 極速發行與金庫代謝防禦 (Rapid Distribution & Protocol L-Archive)
**腳本**：`scripts/maintenance/cloud_archiver.py` / `telegram_manager_bot.py`
**戰術**：
- **CheatSheet 極速發行 (CEO 安全操作)**：廢除高風險、易遭平台封鎖的 RPA 幽靈上傳。CEO 直接開啟生成的 CheatSheet 完美文案，於 YouTube 及 DistroKid 進行手動複製貼上發布，實現零風險、100% 合規的全球上架。
- **100 首定額金庫與 Telegram 預警**：冷金庫 (`ceo_archived_beats`) 嚴守 100 首容量鐵律。達 90% 水位時，透過 Telegram 發送警報，等待 CEO 點擊按鈕授權清理。
- **加權先進先出代謝 (Weighted FIFO)**：清理時，庫存強制降至 50%。系統物理刪除最古老的檔案，但賦予「高衍生次數」的強大基因母帶豁免權（免疫刪除）。衍生歌曲 (is_derivative=True) 達退役標準時直接消滅，僅保留原生母帶進入冷金庫，確保 SSD 空間永不枯竭。



### E1. 數位資產與財務防禦 (Zero-Risk Distribution Rule)
- **廢除高風險 RPA 自動上傳**：為 100% 規避 DistroKid 的 Cloudflare 反爬蟲機制與保護珍貴的版稅帳號，系統已全面淘汰 Playwright 幽靈機器人代操。
- **CheatSheet 絕對控制權**：發行作業回歸「一站式商業企劃合體 (Inline CheatSheet)」。系統負責在後台將音樂、影像、與修飾過的 Tracklist 完美打包，CEO 僅需透過手機或電腦，手動將現成的文案複製貼上至平台，達到「產線極限自動化，發行極限安全化」的最高商業級別。

### E2. 視覺母帶保護 (Static Camera Rule)
- 在呼叫任何 AI 生成微動態影片素材 (I2V) 時，運鏡參數必須強制設定為「靜止 (Static)」。嚴禁任何鏡頭推進或平移，否則 Phase 5 萬能影片處理器在進行 FFmpeg 正反播無縫循環 (Ping-Pong Seamless) 時，將產生嚴重的閃爍與視覺破綻。

### E3. 基礎穩定性防線 (Infrastructure Defenses)
- **原子寫入 (Atomic Writes)**：所有的 JSON 企劃書 (`metadata_distrokid.json`) 與資料庫狀態更新，必須使用原子寫入 (`.tmp` → `os.replace()` 或 DB Commit)。嚴防斷電導致的 0KB 靜默損毀。
- **指令注入與 FFmpeg 語法鐵律**：FFmpeg 與 FFprobe 的調用嚴禁使用 `shell=True`，必須以陣列形式傳遞參數。此外，FFmpeg 濾波圖中的數學表達式（如過場的 `volume` 遮罩）**必須強制使用單引號 `'` 包覆**，防止解析崩潰。

### E4. 零容忍靜默失敗與誠實回報 (No Silent Failure & Absolute Integrity)
- **嚴禁靜默失敗**：程式碼中嚴禁使用空 `try-except` 或僅印出警告卻繼續執行的邏輯。若核心流程 (如選曲引擎、FFmpeg 縫合) 發生異常，必須強制呼叫 `sys.exit(1)` 並立即透過 Telegram 報警。寧可產線停擺，也絕不允許產生「空資產」或「假成功」。
- **資料庫真實性**：系統必須誠實反映歌曲的使用狀態。選中歌曲後必須立刻寫回 `derivation_count + 1`，若發生庫存不足，嚴禁強行拼湊短片，必須觸發備援或等待 CEO 空投。
