# 系統總體架構白皮書：R&S Echoes v15.4 (三引擎智能供彈 · 極速矩陣版)

**版本**：v15.4-Tri-Engine-Matrix  
**最後更新**：2026-04-14（post-release patch 同步）  
**目標環境**：Windows 10 開發機（現役）→ Mac mini M4（24/7 部署目標）

**核心戰略升級（v15.3 → v15.4）**：
- **三引擎 LLM 架構 (Tri-Engine LLM)**：廢除「單一 Gemini」策略。新增 **MiniMax M2.7 230B MoE（預設）**、保留智譜 GLM-4（快速批量）與 Gemini 2.5 Flash（視覺解構備用），三引擎各司其職，透過 `llm_client.py` 統一路由。
- **殭屍行程防護 (Zombie Process Recycling)**：`backend.py` 以 `atexit` 追蹤所有 `Popen` 子程序，Streamlit 意外退出時自動優雅終止孤兒程序。
- **即時日誌流 (Real-time Log Streaming)**：`PYTHONUNBUFFERED=1` 注入子程序環境，解除 Python block buffer 導致 log 長期 0-byte 的問題，UI 即時監控後台進度。
- **Tab4 Final Exports 面板升級**：Step 2（幻影輪播）完成後自動觸發 `st.rerun()`，動態顯示缺失文件並提供一鍵補生成（Tracklist / YouTube CheatSheet / DistroKid CheatSheet）。
- **提示詞批量精簡**：`generate_ceo_prompts.py` 預設由 20 組降為 **5 組**，UI 加入可調組數控件與 MiniMax 耗時預估（每組 ~100s）。
- **SQLite 跨執行緒防護**：`vault_database.py` 改用 `threading.local()` 為每條執行緒建立獨立連線，根除 `sqlite3.ProgrammingError`。廢除 `_get_connection()` 方法，統一改用 `.conn` property；`cloud_archiver.py` 同步修正。
- **SFX 混音比率調整**：`lofi_assembler.py` SFX 音量從 2% 調升至 **4%**。

**v15.4 Post-Release Patch**：
- **歌曲重複次數預設值調升（3 → 5）**：`lofi_assembler.py` / `backend.run_phase4_sequence()` / UI 側邊欄預設值全部從 3 次改為 **5 次**，延長每首歌的使用壽命、降低 TTAPI 備援觸發頻率。
- **Telegram `/status` 頻道分欄顯示**：重寫戰情報表，按 `🎵 Lofi` / `🌟 Light Music` 分欄列出各頻道的活水 / 歸檔 / 衍生層分布，改用 **HTML parse mode** 避免 MarkdownV2 特殊字元跳脫錯誤。
- **ngrok Authtoken 支援**：`.env` 新增 `NGROK_AUTHTOKEN`，`start_telegram_bot.py` 在 `connect()` 前自動注入，解決 `ERR_NGROK_4018` 認證失敗問題。

## ▌ 0. 產線鐵律（The Ironclad Rules — 不可違反）

| # | 鐵律 | 對應機制 |
|---|------|---------|
| 1 | **Streamlit 樞紐單一化** | 廢除 `rs_manager.py` CLI；所有操作必須走 `app.py` Web UI |
| 2 | **UI 非阻塞** | 耗時任務以 `threading.Thread(daemon=True)` 背景執行，嚴禁卡死 UI |
| 3 | **絕對音畫同步** | Stage 2 強制 `-t 3600`；廢除 `-shortest`；音軌先在 Stage 2.5 預混音 |
| 4 | **雙棲實體路徑隔離** | `.env` 的 `WORKSPACE_ROOT` + `env_manager.py` 跨平台解析 |
| 5 | **零浪費延遲扣款** | 成片寫入後才執行 `derivation_count + 1`，絕不預扣款 |
| 6 | **多頻道物理隔離** | `--channel lofi / light_music` 嚴格隔離基因、DB 與目錄 |
| 7 | **零容忍靜默失敗** | 核心異常必須 `sys.exit(1)`，嚴禁空 `try-except` 假成功 |
| 8 | **原子寫入防護** | JSON / DB 狀態更新必須 `.tmp` → `os.replace()` 或 DB Commit |
| 9 | **殭屍行程防護** | `atexit` 追蹤所有 `Popen`，Streamlit 退出時自動清理 |
| 10 | **即時日誌流** | 子程序環境注入 `PYTHONUNBUFFERED=1`，log 即時落盤 |

## ▌ 1. 技術棧全覽 (v15.4 現代化對齊版)

### 1.1 大腦協同 — 三引擎 LLM 架構

```
┌─────────────────────────────────────────────────────────┐
│               llm_client.py  統一路由閘道                 │
│  provider="minimax" ──▶ MiniMax M2.7 230B (NVIDIA NIM)  │  ← 預設，複雜推理
│  provider="zhipu"   ──▶ 智譜 GLM-4                      │  ← 快速批量生成
│  provider="gemini"  ──▶ Google Gemini 2.5 Flash          │  ← 視覺解構備用
└─────────────────────────────────────────────────────────┘
```

| 引擎 | 模型 | API 協定 | 速度 | 適用場景 |
|------|------|---------|------|---------|
| MiniMax M2.7 (預設) | `minimaxai/minimax-m2.7` | OpenAI-compatible (NVIDIA NIM) | ~100s/次 | 複雜 Agent、發行文案 |
| 智譜 GLM-4 | `glm-4` | ZhipuAI SDK | ~5-10s/次 | 批量提示詞、快速回應 |
| Gemini 2.5 Flash | `gemini-2.5-flash` | google-generativeai | ~15-30s/次 | 視覺解構、多模態 |

> **延遲提示**：MiniMax M2.7 實測每次 API 呼叫約 100 秒，5 組提示詞總耗時約 8-9 分鐘。批量供彈建議改用智譜 GLM-4。

### 1.2 動態影音處理引擎

- **FFmpeg / FFprobe**：所有影音操作的底層引擎
  - `Encode-Once-Repeat`：短素材標準化 1 次後以 `concat demuxer` 無損複製延長
  - `Pre-baked Fade`：素材鑄造 in/mid/out 三版本，物理拼接實現呼吸燈轉場
  - `Acrossfade Premix`：Stage 2.5 將多首音軌預混音為單一長檔，防 EOF 截斷
  - **Windows 開發機**：`libx264` 軟解；**Mac M4 部署**：`h264_videotoolbox` 硬加速

### 1.3 數位資產代謝系統

- **SQLite3**：雙金庫並行，`threading.local()` 確保跨執行緒安全
  - `rs_music_vault.db`：音樂資產管理（衍生次數、頻道標籤）
  - `veo_visual_vault.db`：視覺素材管理（50/25/25 金字塔抽樣）
- **金字塔抽樣**：50% 全新 / 25% 二手 / 25% 三手 + 三層兜底策略（v15.4 新增）
- **Clip Cycling**：當金庫素材不足但非零時，自動循環複用以避免 Fatal Error

## ▌ 2. API 金鑰管理 (.env 完整清單)

| `TT_BASE_URL` | TTAPI endpoint（預設 `https://api.ttapi.io`）| 選填 |

| `TELEGRAM_ALLOWED_USER_ID` | 授權操作 Bot 的 CEO Telegram User ID | 選填 |
| `TELEGRAM_WEBHOOK_URL` | Webhook 模式公網 HTTPS URL（留空 → 自動退回 Polling）| 選填 |
| `TELEGRAM_WEBHOOK_PORT` | Webhook 本機監聽埠（預設 8443）| 選填 |
| `NGROK_AUTHTOKEN` | ngrok 隧道認證 Token（`--ngrok` 模式必填，免費帳戶即可）| ✅（使用 --ngrok 時）|
| `MJ_API_KEY` / `MJ_BASE_URL` | Midjourney / TTAPI（視覺生成）| 選填 |
| `KLING_AK` / `KLING_SK` | Kling AI（視覺生成）| 選填 |
| `AWS_S3_BUCKET` 等 | AWS S3 冷儲存 | 選填 |

#### 4.3.4 `_run_suno_backup()` — TTAPI 自動供彈詳解

```
_run_suno_backup(target_count=5, channel="lofi")
│
├─ 查 ceo_approved_beats/{ch}/*.mp3 數量
│   ≥ target_count(5) → return True（庫存充足）
│
└─ 不足 → needed = target_count - count
    │
    ├─ ① Protocol L 衍生（優先，省成本）
    │   從 rs_music_vault.db 取 derivation_count < 3 的曲目
    │   輪流套用 tempo_up / tempo_down / pitch_up / pitch_down
    │   → 衍生成功 → 落入 ceo_approved_beats/{ch}/
    │   → needed -= generated_count
    │   若 needed ≤ 0 → 母帶化 → 入庫 → return True
    │
    └─ ② 仍不足 → TTAPI 新歌生成
        │
        ├─ 讀取配方 assets/.ceo_prompts/daily_prompts_{CH}_*.txt
        │   若不存在 → 呼叫 generate_ceo_prompts.py（LLM 即時生成）
        │
        └─ 循環 needed 次：
            suno_api_engine.py \
              --prompt  "..." \
              --tags    "lofi, ..." \
              --title   "Track_XXX" \
              --output-dir ceo_approved_beats/{ch}/ \
              --channel lofi
            timeout=600s（10 分鐘 per track）
            成功 → audio_mastering_engine.py → vault DB 入庫
            失敗 → 記錄，繼續下一首
```

**備援觸發常數**：
| 常數 | 值 | 說明 |
|------|---|------|
| `target_count` | 5 | 備援目標曲數（ceo_approved_beats） |
| `MINIMUM_INVENTORY_FOR_PRODUCTION` | 10 | 庫存警告閾值（vault_ready_for_mix）|
| `timeout per track` | 600s | Suno API 單首生成超時 |

> **注意**：`_run_suno_backup()` 檢查 `ceo_approved_beats/` 的 `.mp3` 數量作為觸發條件，而 `lofi_assembler.py` 查的是 `vault_ready_for_mix/` 的 WAV。兩個目錄的歌曲狀態有 Phase 差距：新生成的歌先落在 `ceo_approved_beats/`，經母帶化後才進入 `vault_ready_for_mix/`。

```python
# threading.local() 確保每條執行緒有獨立的 sqlite3 連線
# ⚠️ _get_connection() 已廢除，所有呼叫端一律改用 .conn property
_local = threading.local()

@property
def conn(self) -> sqlite3.Connection:
    c = getattr(self._local, "conn", None)
    if c is None:
        c = sqlite3.connect(str(self.db_path), timeout=30)
        c.row_factory = sqlite3.Row
        self._local.conn = c
    return c
```
