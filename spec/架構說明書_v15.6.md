# 系統總體架構白皮書：R&S Echoes v15.6 (廣播級無縫矩陣引擎)

**版本**：v15.6-Broadcast-Grade-Seamless  
**最後更新**：2026-04-15（v15.6 Root Cause Fix 同步）  
**目標環境**：Windows 10 開發機（現役）→ Mac mini M4（24/7 部署目標）

**核心戰略升級（v15.4 → v15.6）**：

**v15.5 — 無縫迴圈廣播級防禦（由外部顧問 RCA + CTO + 工程部聯合修復）**：
- **Closed GOP (`-flags +cgop`)**：強制封閉 GOP，確保 `base_mid.ts` 結尾的 B 幀絕不跨界依賴下一個 TS 的 I 幀，從根本消除 B 幀解碼斷層
- **嚴格 IDR 鎖定 (`-g 48 -keyint_min 48 -sc_threshold 0`)**：最大與最小 GOP 長度同時鎖定為 48 幀，關閉動態場景偵測，讓每 2 秒精確出現一個 I 幀，concat 邊界永遠完美對齊
- **濾鏡執行順序修正**：`setpts=PTS-STARTPTS,fps=24` 必須在 `fade` **之前**執行。先將 AI 素材的「髒時間軸」歸零至絕對 0-based，再讓 fade 的 `st=` 參數落在乾淨的 24fps 時間軸上，達到幀級精準
- **統一時間基底 (`-video_track_timescale 24000`)**：以整數算術統一所有片段的時間戳單位，防止 24fps 浮點誤差在 400+ 片段中累積崩塌
- **GOP-QA 自動驗證**：新增 `_verify_gop_closure()` 方法，每次 Stage 1 編碼後自動清點 `base_mid.ts` 的 I 幀間隔，日誌即時報告 GOP 牢籠是否完好
- **音訊漂移防禦 (`-ar 48000`)**：Stage 2 最終 AAC 輸出強制 48000Hz，從根源消除 44100↔48000Hz 浮點換算在長片中積累的音畫漂移

**v15.6 — 根治「8 秒凍結」真正元凶（工程部 RCA，與顧問英雄所見略同）**：
- **`get_video_duration()` 源幀率修正**：原版硬編碼 `frames / 24.0`，但生產素材全為 **30fps**（240 幀）。錯誤計算 `240 ÷ 24 = 10s`，導致 `loop_list.txt` 宣告 `duration 10`，而 `base_mid.ts` 實際只有 8 秒，Concat demuxer 播完 8 秒後等待 2 秒空洞，造成畫面凍結。修復：ffprobe 同步讀取 `r_frame_rate`，改用 `frames / source_fps`（`240 ÷ 30 = 8.0s`）
- **雙重保險 re-probe**：3-Pass 編碼完成後，直接清點 `base_mid.ts` 的實際輸出幀數（已強制 fps=24），以 `ts_frames / 24.0` 作為 `loop_list` 的 `duration` 宣告值，消除 fps 轉換捨入的最後一個誤差源，實現幀級吻合零空洞

## ▌ 1. 技術棧全覽 (v15.6 對齊版)

### 1.1 大腦協同 — 三引擎 LLM 架構

```
┌─────────────────────────────────────────────────────────┐
│               llm_client.py  統一路由閘道                 │
│  provider="minimax" ──▶ MiniMax M2.7 230B (NVIDIA NIM)  │  ← 預設，複雜推理
│  provider="zhipu"   ──▶ 智譜 GLM-4                      │  ← 快速批量生成
│  provider="gemini"  ──▶ Google Gemini 2.5 Flash          │  ← 視覺解構備用
└─────────────────────────────────────────────────────────┘
```

| 引擎 | 模型 | API 協定 | 速度 | 適用場景 |
|------|------|---------|------|---------|
| MiniMax M2.7 (預設) | `minimaxai/minimax-m2.7` | OpenAI-compatible (NVIDIA NIM) | ~100s/次 | 複雜 Agent、發行文案 |
| 智譜 GLM-4 | `glm-4` | ZhipuAI SDK | ~5-10s/次 | 批量提示詞、快速回應 |
| Gemini 2.5 Flash | `gemini-2.5-flash` | google-generativeai | ~15-30s/次 | 視覺解構、多模態 |

> **延遲提示**：MiniMax M2.7 實測每次 API 呼叫約 100 秒，5 組提示詞總耗時約 8-9 分鐘。批量供彈建議改用智譜 GLM-4。

### 1.2 動態影音處理引擎（v15.6 廣播級標準）

- **FFmpeg / FFprobe**：所有影音操作的底層引擎
  - `Encode-Once-Repeat`：短素材標準化 1 次後以 `concat demuxer` 無損複製延長
  - `Pre-baked Fade`：素材鑄造 in/mid/out 三版本，物理拼接實現呼吸燈轉場
  - `Closed GOP`：`-flags +cgop -g 48 -keyint_min 48 -sc_threshold 0`，廣播級無縫拼接標準
  - `PTS 歸零優先`：`setpts=PTS-STARTPTS,fps=24` 永遠在 fade 濾鏡之前執行
  - `Acrossfade Premix`：Stage 2.5 將多首音軌預混音為單一長檔，防 EOF 截斷
  - **Windows 開發機**：`libx264` 軟解；**Mac M4 部署**：`h264_videotoolbox` 硬加速

### 1.3 數位資產代謝系統

- **SQLite3**：雙金庫並行，`threading.local()` 確保跨執行緒安全
  - `rs_music_vault.db`：音樂資產管理（衍生次數、頻道標籤）
  - `veo_visual_vault.db`：視覺素材管理（50/25/25 金字塔抽樣）
- **金字塔抽樣**：50% 全新 / 25% 二手 / 25% 三手 + 三層兜底策略
- **Clip Cycling**：當金庫素材不足但非零時，自動循環複用以避免 Fatal Error

```python
# threading.local() 確保每條執行緒有獨立的 sqlite3 連線
# ⚠️ _get_connection() 已廢除，所有呼叫端一律改用 .conn property
_local = threading.local()

@property
def conn(self) -> sqlite3.Connection:
    c = getattr(self._local, "conn", None)
    if c is None:
        c = sqlite3.connect(str(self.db_path), timeout=30)
        c.row_factory = sqlite3.Row
        self._local.conn = c
    return c
```

**v15.6 廣播級編碼防禦陣列（鎖定，禁止修改）：**

```python
vf_pre = "setpts=PTS-STARTPTS,fps=24"  # 【順序鎖定】PTS 歸零在 fade 之前
common_vf_base = f"{vf_pre},{norm}"

common_enc = [
    "-map_metadata", "-1", "-an",
    "-c:v", "libx264", "-preset", "medium", "-tune", "film", "-crf", "28",
    "-pix_fmt", "yuv420p",
    "-g", "48", "-keyint_min", "48",    # 嚴格 IDR：最大 = 最小 = 48 幀
    "-sc_threshold", "0",               # 關閉動態 I 幀偵測
    "-flags", "+cgop",                  # 強制封閉 GOP
    "-video_track_timescale", "24000",  # 統一時間基底
]

passes = [
    (base_in,  f"{vf_pre},fade=t=in:st=0:d={fade_dur},{norm}",              "fade-in"),
    (base_mid, common_vf_base,                                               "clean"),
    (base_out, f"{vf_pre},fade=t=out:st={fade_out_st}:d={fade_dur},{norm}", "fade-out"),
]
```

**濾鏡執行順序說明（核心設計決策）：**
```
❌ 錯誤順序: fade → setpts → fps=24 → norm
   問題：fade 的 st= 參照 AI 原始「髒時間軸」（可能從 0.041s 開始），淡出錯位

✅ 正確順序: setpts → fps=24 → fade → norm
   原理：先建立絕對乾淨的 0-based 24fps 時間軸，fade 的 st= 才能精準落點
```

#### 4.7.3 `get_video_duration()` — v15.6 源幀率修正

```python
# 同步讀取 nb_read_packets + r_frame_rate，一次 ffprobe 完成
# 公式：duration = frames / source_fps（非硬編碼 24.0）
#
# 正確示例：
#   30fps 素材 240 幀 → 240 / 30 = 8.0s ✅
#   24fps AI 素材（標頭誤寫 10s）192 幀 → 192 / 24 = 8.0s ✅
#
# 舊版錯誤：
#   30fps 素材 240 幀 → 240 / 24 = 10.0s ← 造成 loop_list duration=10，
#   實際 .ts 只有 8s → Concat demuxer 等待 2 秒空洞 → 畫面凍結
```

#### 4.7.4 雙重保險 re-probe（v15.6 新增）

```python
# 3-Pass 編碼完成後，直接清點 base_mid.ts 的實際輸出幀數

ts_frames = ffprobe(base_mid, nb_read_packets)
```
